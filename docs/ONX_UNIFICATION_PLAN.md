# ONX Unification Plan — خطة توحيد حضارة ONX

> **الإصدار:** v1.0 · **التاريخ:** 2026-07-07 · **الحالة:** مسوّدة تنفيذية للاعتماد
> **المرجعيات الحاكمة:** `docs/ONX_MASTER_EXECUTION_DOCUMENT_v3.0.md` (MED v3.0) · `docs/BRIDGE_INTEGRATION_CONTRACT.md` (عقد الجسر v1.0)
> **الغرض:** تحويل عائلة مستودعات ONX المتفرّقة (3+1) إلى **مشروع واحد موحّد** قابل للاختبار والنشر — مع الحفاظ على العزل الدستوري — تمهيداً لترحيل AWS بعد اجتياز بوابات النشر.

---

## 0. الخلاصة التنفيذية

مشروع ONX **مشروع واحد بمسارين**: **أمنكس Platform** (الجسد التشغيلي) و**أونكس Intelligence** (العقل الدستوري). اليوم هما موزّعان على أربعة مستودعات منفصلة بحالات نضج متفاوتة. هذه الخطة تدمجهما في **مستودع أحادي (monorepo) واحد** بحدود صارمة، بحيث:

1. **الدمج بنيوي (repository consolidation) — وليس انصهاراً في وقت التشغيل.** يبقى المساران حزمتين قابلتين للنشر منفصلاً، ويتواصلان **حصراً** عبر الجسر المؤمّن (عقد الجسر v1.0). هذا يحفظ **قفل V6 الدستوري**.
2. **مصدر حقيقة واحد للدستور:** المستودع الدستوري المفقود (`onx-constitutional-assets`، حالياً HTTP 404) يُنشأ كحزمة جذر `packages/constitution` يستوردها المساران للقراءة فقط.
3. **مسار ترحيل تدريجي ببوابات:** كل مرحلة لها دليل إثبات وبوابة اعتماد، تنتهي بـ **بوابة النشر 3 = AWS** (مؤجّلة بقرار المؤسس حتى اجتياز التشغيل التجريبي والاختبار الكامل).

**المبدأ الحاكم:** *نوحّد المصدر (source of truth) دون أن نكسر الحدود (boundaries).* الاندماج في Git لا يعني اندماجاً في الصلاحيات.

### 2.1 قواعد الحدود (Boundary Rules) — غير قابلة للتفاوض

| القاعدة | الفرض التقني |
|---|---|
| `apps/platform` **لا يستورد** كوداً من `apps/intelligence` مباشرةً | `dependency-cruiser` / ESLint `no-restricted-imports` يفشل البناء عند الخرق |
| التواصل بين المسارين **حصراً** عبر HTTP + `packages/bridge-contract` | مراجعة معمارية + اختبار عقد |
| `packages/constitution` **للقراءة فقط** — لا يستورد من `apps/*` | اتجاه اعتماد أحادي (apps → packages، never reverse) |
| قفل V6 يبقى علَم تشغيل (`BRIDGE_ENABLED=false`) بعد الدمج | لا يتغيّر سلوك الجسر بمجرد وضعهما في ريبو واحد |

> **جوهر التصميم:** وضع المسارين في مونوريبو واحد يمنحنا **مصدر حقيقة + بناء/اختبار موحّد + إصدار ذرّي**، بينما تمنع قواعد الحدود أي انصهار غير مأذون. الدمج الفيزيائي لا يفكّ القفل الدستوري.

## 3. آلية الدمج الفيزيائي (Git History Preservation)

نستخدم `git subtree` (أو `git filter-repo`) لإدخال كل مستودع تحت مجلده مع **الحفاظ الكامل على التاريخ**:

```bash
# داخل مونوريبو جديد onx/
git subtree add --prefix=apps/intelligence   <intelligence-remote> main
git subtree add --prefix=apps/platform        <platform-remote>     main
git subtree add --prefix=packages/knowledge   <platform-complete-remote> main
# constitution يُنشأ يدوياً (repo #4 غير موجود) ثم يُضاف
```

- **لماذا subtree لا submodule:** نريد شجرة عمل واحدة قابلة للبناء الذرّي، لا مؤشرات هشّة.
- **الأصول الثقيلة** (`onx-database.db` 9MB، `knowledge-seed-15k.json` 11MB): تُدار عبر **Git LFS**.
- **تعارضات الأسماء** (كلا المسارين فيه `api/`, `src/`): محلولة تلقائياً لأن كلاً منهما يسكن تحت `apps/<name>/` منفصل.

## 4. خطة الترحيل بالمراحل (Phased Migration)

المراحل تنسجم مع خطوط MED v3.0 الأربعة: **C** (دستور) · **I** (ذكاء) · **P** (منصة) · **B** (جسر)، وتنتهي ببوابات النشر الثلاث.

### المرحلة 0 — التأسيس الدستوري (الخط C) — *حاجزة لكل ما بعدها*
- [ ] إنشاء `onxos/onx-constitutional-assets` (خاص) → يردّ 200 بدل 404.
- [ ] إيداع الأصول الـ305 (33 كابستون + 131 SBP + 85 ملف MCR-03 + 12 برنامجاً جذرياً + الجذور الثمانية) بجرد موقّع بسلسلة hash.
- [ ] ربط سلسلة النسب (كل أصل → جذره → مبدؤه) وإعادة تشغيل CIV-02 حتى PASS.
- **بوابة C:** الدستور حيّ ومُتحقَّق، لا أصل يتيم.

### المرحلة 1 — إنشاء هيكل المونوريبو
- [ ] مونوريبو `onx` بـ workspaces + Turborepo + `tsconfig.base.json` صارم.
- [ ] استيراد `packages/constitution` + `packages/bridge-contract` (استخراج المخططات الحالية من `api/bridge-guard.ts` + `titan-bridge-router.ts` + `corpus-query-router.ts` + `intent-engine-router.ts`).
- **بوابة 1أ:** `pnpm build` على الحزم المشتركة ينجح بصفر أخطاء TS.

### المرحلة 2 — إدخال المسارين (subtree)
- [ ] `apps/intelligence` (هذا المستودع) → البناء + الـ13 ملف اختبار يمرّون كما هم.
- [ ] `apps/platform` (Atlas V5) → البناء نظيف (كما في MED v3.0).
- [ ] `packages/knowledge` + `infra/aws`.
- **بوابة 2أ:** بناء كل حزمة مستقلاً أخضر · لا استيراد عابر للحدود (dependency-cruiser أخضر).

### المرحلة 3 — توصيل الحدود المشتركة
- [ ] `apps/intelligence` يستورد الدستور من `packages/constitution` بدل النسخ المحلية.
- [ ] الطرفان يستوردان مخططات الجسر من `packages/bridge-contract` (مصدر واحد).
- [ ] `apps/platform` (titanBridge) يوجَّه للطرف المقابل في `apps/intelligence` — لكن **يبقى معطّلاً بـ V6**.
- **بوابة 3أ:** اختبار عقد الجسر (`bridge-contract.test.ts`) أخضر من الجانبين.

### المرحلة 4 — التشغيل التجريبي الموحّد (= بوابة النشر 1)
- [ ] تشغيل تجريبي شامل (كالموجود: `api/__tests__/trial-run.test.ts` — دورة حياة كاملة عبر المحرّكات).
- [ ] توسعته ليعبر حدّ المنصة↔الذكاء عبر الجسر (بسرّ اختبار، بيئة معزولة).
- [ ] فحص صحة حيّ لكل خدمة (`/api/trpc/health.ping` + `health.status`).
- **بوابة النشر 1 (تجريبي):** كل المحرّكات تعمل معاً في CI + صحة حيّة خضراء.

### المرحلة 5 — الاختبار الكامل للنظام (= بوابة النشر 2)
- [ ] تغطية اختبار شاملة للمساري (وحدة + E2E + عقد + حِمل).
- [ ] إغلاق دَين الجودة المتبقّي (تغطية DB-dependent D11–D20 عند توفّر MySQL في بيئة اختبار).
- [ ] تدقيق أمني للجسر (اختبار سلبي: بلا ترويسة → `BRIDGE_UNAUTHORIZED`).
- **بوابة النشر 2 (اختبار كامل):** النظام الموحّد مُختبَر من طرف لطرف.

### المرحلة 6 — ترحيل AWS (= بوابة النشر 3، بتوقيع المؤسس)
- [ ] تفعيل `infra/aws` (Terraform من repo #3): VPC + RDS (Postgres/MySQL) + ECS Fargate لكل خدمة + ALB + Secrets Manager (لسرّ الجسر).
- [ ] خطّان مستقلان للنشر: `intelligence-service` و`platform-service` خلف نفس الـ VPC، يتواصلان داخلياً عبر الجسر.
- [ ] نقل بذور المعرفة + قاعدة البيانات إلى RDS/S3.
- **بوابة النشر 3 (AWS):** الحضارة الموحّدة حيّة على AWS · الجسر قابل للتفعيل بموافقة V6.

## 5. تسلسل البوابات (Gate Sequence)

```
C (دستور حيّ) ─► المونوريبو ─► إدخال المسارين ─► توصيل الحدود
      └────────────────────────────────────────────────┘
                          │
             بوابة النشر 1: تشغيل تجريبي  ✅ (أساسها جاهز: trial-run.test.ts)
                          │
             بوابة النشر 2: اختبار كامل للنظام
                          │
             بوابة النشر 3: AWS  (توقيع المؤسس — قرار 2026-07-07)
```

**قرار المؤسس النافذ:** ترحيل AWS **مؤجّل** حتى اجتياز البوابتين 1 و2. لا نبني بنية AWS قبل أن يُثبت النظام الموحّد نفسه تشغيلياً واختبارياً.

## 6. المخاطر والتخفيف

| الخطر | الأثر | التخفيف |
|---|---|---|
| كسر الحدود بعد الدمج (استيراد مباشر عابر) | انهيار العزل الدستوري (V6) | فرض آلي عبر `dependency-cruiser` في CI — يفشل البناء |
| المستودع الدستوري ما يزال 404 | لا سلطة حقيقية للدمج | المرحلة 0 حاجزة — لا دمج قبل دستور حيّ |
| تضخّم المونوريبو (أصول ثقيلة) | بطء الاستنساخ | Git LFS للأصول > 5MB |
| ازدواج مخططات الجسر بين الطرفين | انحراف العقد | `packages/bridge-contract` مصدر واحد يستورده الطرفان |
| فقدان تاريخ Git عند الدمج | ضياع سلسلة المسؤولية | `git subtree` يحفظ التاريخ الكامل تحت المجلدات |

## 7. الإجراءات الفورية (الأسبوع الأول)

1. **قرار المؤسس:** اعتماد نمط المونوريبو + قاعدة الحدود (هذه الوثيقة).
2. **إنشاء `onx-constitutional-assets`** وبدء إيداع الأصول الـ305 (المرحلة 0).
3. **استخراج `bridge-contract`** كحزمة مشتركة من الكود الحالي (خطوة غير كاسرة، قابلة للتنفيذ في هذا المستودع فوراً).
4. **تجهيز سكربت المونوريبو** (`git subtree add` للمستودعات الثلاثة الموجودة) في فرع تجريبي.
5. **الإبقاء على النشر الحالي** (Render، فرع `main` للإنتاج، `onxos-fix-render-build-deps` للـstaging) دون تعطيل حتى تكتمل البوابات.

## 8. الإصدار وإدارة التغيير

- هذه الوثيقة **v1.0**. أي تغيير في نمط المونوريبو أو قواعد الحدود يرفع النسخة الكبرى ويتطلب توقيع المؤسس.
- تكمّل هذه الخطة عقد الجسر v1.0 ولا تنسخه: العقد يحكم *كيف يتواصل* المساران؛ هذه الخطة تحكم *كيف يسكنان* مستودعاً واحداً دون أن يندمجا في الصلاحيات.
- المصدر الأوثق لحقائق المستودعات: MED v3.0 القسم 0.
